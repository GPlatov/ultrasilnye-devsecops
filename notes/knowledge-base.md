# Knowledge Base — День 1 (Фундамент)

## 1) Репозиторий (Repository)
**Репозиторий** — это папка проекта + история изменений.
Git хранит историю в скрытой папке **.git**.

## 2) Коммит (Commit)
**Коммит** — это сохранённый “снимок” изменений.
В коммите есть: автор, время, сообщение и список изменений.

## 3) Ветка (Branch) и main
**Ветка** — отдельная линия изменений.
**main** — главная ветка проекта (то, что считается “основной версией”).

## 4) Git status (что я сегодня увидел)
Команда git status показала:
- On branch main
- nothing to commit, working tree clean
Это значит: я в ветке main и нет изменений, которые нужно сохранять.

## 5) CI (Continuous Integration) — простыми словами
**CI** — автоматические проверки проекта после того, как я отправил изменения на GitHub.
Цель CI: быстро поймать ошибки и проблемы безопасности.
Если CI зелёный ✅ — можно двигаться дальше.
Если CI красный ❌ — надо исправлять.
### 6) Security Gates (ворота безопасности)
Security Gates — это проверки в CI, которые НЕ пропускают опасные изменения в репозиторий.

В моём проекте сейчас стоят ворота:
1) Gitleaks — ищет секреты (токены/ключи). Нашёл → CI становится красным.
2) Trivy — ищет уязвимости уровня HIGH/CRITICAL. Нашёл → CI становится красным.

Идея: безопасность встроена в процесс, а не “потом”.

### 7) Remote / origin / push / pull (по-детски)
Remote — это удалённая копия репозитория (на GitHub).
origin — имя по умолчанию для remote.

git push — отправить мои коммиты на GitHub (в origin).
git pull — забрать свежие изменения с GitHub на мой компьютер.
## 8) SAST и CodeQL (простыми словами)

**SAST** — это поиск уязвимостей в коде без запуска программы.

**CodeQL** — это встроенный инструмент GitHub для SAST.
Он анализирует репозиторий и ищет потенциально опасные места в коде.

Где это видно:
- вкладка **Security**
- раздел **Code scanning**
- вкладка **Actions**, где запускается workflow **CodeQL**

Что я сделал:
- включил **CodeQL analysis**
- выбрал режим **Default setup**
- GitHub автоматически добавил и запустил проверку

Зачем мне это нужно:
- ловить проблемы в коде заранее
- усиливать безопасность проекта
- строить портфолио Security Engineer / DevSecOps
## 9) Permissions в GitHub Actions (простыми словами)

**Permissions** — это права, которые получает workflow в GitHub Actions.

Если permissions не указать явно, workflow может получить права по умолчанию, которые шире, чем реально нужны.

Безопасный подход:
- давать только те права, которые действительно нужны
- не оставлять лишние разрешения “на всякий случай”

CodeQL нашёл у меня предупреждение:
"Workflow does not contain permissions"

Это значит:
в файле `.github/workflows/ci.yml` нужно явно указать блок permissions.
